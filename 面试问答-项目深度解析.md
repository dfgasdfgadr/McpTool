# 面试官深度问答：AI 请求智能分析助手（扩展 + MCP）

> 本文档按「模拟面试官提问」组织，结合本仓库实现作答；用于面试准备、述职或新人答疑。部分「项目背景」表述需你按真实业务/立项原因替换为事实描述。

---

## 1. 这个项目的背景是什么？

### 可回答的「问题背景」（技术侧）

在 Web 研发与调试里，大量真实接口信息存在于**浏览器会话与 Network 面板**中，而不是及时的 OpenAPI 文档里。团队常见痛点包括：

- **文档滞后**：页面已上线，文档未更新，开发和测试只能靠 DevTools「人肉读包」。
- **AI 编排缺契约**：希望用 Cursor 等客户端通过 **MCP** 调用业务接口，但手写工具定义（URL、方法、Schema、鉴权头）成本高、易与真实流量不一致。
- **调试依赖后端**：联调与演示阶段需要改响应、Mock、重复构造数据，纯前端验证链路长。

### 本项目的定位（一句话）

做一个**挂在真实页面上的扩展**，把 XHR/Fetch 流量变成**结构化记录**，可选 **AI 语义分析**，并进一步**生成 MCP 工具**，通过**本地助手进程**对接到 MCP 客户端；同时提供 **Mock/改写** 缩短调试路径。

### 提醒你补充的「事实背景」（面试常追问）

- **立项动机**：个人效率工具 / 团队内部平台 / 大赛或课程项目？  
- **服务对象**：仅自己用还是团队共用的规范（配置、合规）？  
- **成功标准**：例如「生成 MCP 替代手写封装」「分析一条请求从 X 分钟到 Y 秒」——有数据更好。

## 2. 你是怎么接入 AI（大模型）的？

### 实现路径（与代码一致）

1. **配置在扩展侧**：用户在配置面板填写 **Base URL、API Key、模型名**（见 `state` + `ui-core` 配置逻辑），本质是 **OpenAI Chat Completions 兼容**的形态（路径拼接为 `{baseURL}/chat/completions`）。
2. **内容脚本不发直连外网**：`content/ai-analysis.js` 中的 `callAI` 使用 `chrome.runtime.sendMessage` 发 **`AI_CHAT_COMPLETIONS`**。
3. **由 Service Worker 代发 HTTP**：`background.js` 中监听该消息，用 **`fetch` + `Authorization: Bearer {apiKey}`** 调用远程接口，解析 `choices[0].message.content` 回传给内容脚本。
4. **业务封装**：单条分析与批量串联在 `analyzeRequest` / `analyzeRequestsSequential`，Prompt 中拼接方法、URL、请求体与响应体（先做截断以降低 token），写回记录的 `aiAnalysis` 字段并刷新 UI。

### 这样设计的原因

- **密钥不暴露在页面上下文**：密钥留在扩展存储与后台，降低被页面其它脚本窃取的风险（仍需注意扩展本身被打包篡改等风险）。
- **统一出口**：可做限流、错误包装、日志（若后续要加）。
- **兼容多云**：只要 API 形态兼容 OpenAI Chat Completions，即可换厂商。

### 可能的追问与补充

- **流式输出**：当前实现为一次性 `content` 文本，未在代码中体现 SSE 流式。  
- **隐私与合规**：分析内容可能含 PII/密钥，是否脱敏、是否走内网模型，需在真实项目中说明。

## 3. 客户端与扩展是通过什么通信的？为什么能互通？

这里要分清 **「谁和谁是客户端」**——本方案里至少有三段链路。

### 3.1 MCP 客户端（如 Cursor）↔ 本地 `mcp-helper`（Node）

- **通道**：**HTTP**（本仓库中助手在本地端口例如 9527 上处理 **`/mcp`** 等路径），协议为 **JSON-RPC** 风格的 MCP 消息（`initialize`、`tools/list`、`tools/call` 等，见 `extension/mcp-helper/server.mjs`）。
- **为什么能通**：IDE 把 MCP Server 配成 `http://127.0.0.1:端口/mcp`，走本机回环，**无浏览器跨域问题**；助手进程独立监听端口，不依赖网页。

### 3.2 `mcp-helper`（Node）↔ 浏览器扩展（Service Worker）

- **通道**：**Chrome Native Messaging**。扩展侧 `chrome.runtime.connectNative('com.aireq.mcp_helper')` 拉起本机注册的 Host，双方在 **stdin/stdout** 上传输 **4 字节小端长度 + UTF-8 JSON**（见 `server.mjs` 中 `readNMMessage` / `writeNMMessage`）。
- **为什么能通**：Chrome 提供 **Native Messaging** 契约：本机安装 manifest 描述的可执行文件，扩展与本地进程可双向发消息；**绕开网页安全模型**，也**不要求**扩展开放在网页里监听端口。
- **消息类型**：如 `PING`/`PONG` 握手、`SYNC_TOOLS` 同步工具列表、`CALL_REQUEST`/`CALL_RESULT` 把 MCP 的工具调用搬进扩展世界里执行。

### 3.3 Service Worker ↔ 页面里的 Content Script

- **通道**：`chrome.runtime.sendMessage`、`chrome.tabs.sendMessage`。
- **为什么能通**：同属一个扩展 ID，Chrome 运行时负责路由；常用于「后台编排、前台执行」。
- **典型场景**：后台收到 `CALL_REQUEST` 后，希望对目标站发起**带页面 Cookie** 的请求：先 `tabs.query` 找匹配 **origin** 的标签页，再通过 **`chrome.tabs.sendMessage` 派发 `MCP_PROXY_REQUEST`** 到内容脚本。

### 3.4 Content Script（隔离世界）↔ 页面主世界（PAGE）

- **通道**：`window.postMessage`；主世界脚本由 **`chrome.scripting.executeScript`** 注入 `page-hook.js`（`world: 'MAIN'`），与 Hook 的安装检测（`READ_PAGE_HOOK_INSTALLED`）配合。
- **为什么能通**：内容脚本与原页面 JS 内存隔离，**不能直接改**页面里的原生 `fetch`；通过 **注入 MAIN world** + **postMessage** 把抓取到的记录送回隔离世界，再 `addRequestRecord`。

### 小结（面试一句话）

**IDE ↔ 本地 HTTP MCP；本地进程 ↔ 扩展 ↔ 标签页内容脚本；内容脚本 ↔ 页面 Hook 用 postMessage。** 各段选用不同机制，是因为**安全模型、Cookie 携带、以及扩展无法被网页直接当服务器访问**等约束。

## 4. 文件结构如何划分？有哪些模块、关联与作用？

下列按 `extension/manifest.json` 中 **content_scripts 顺序** + 关键文件说明（**加载顺序有依赖**）。

| 模块 / 文件 | 作用 | 与其它模块的关联 |
|-------------|------|------------------|
| **`content/state.js`** | 全局状态：请求列表、配置、MCP 工具、UI 状态等 | 被所有后续脚本读取/修改 |
| **`content/mock-rules.js`** | Mock / 调试规则存储与匹配 | `core` 打请求时消费；与页面同步规则 |
| **`content/core.js`** | 请求拦截（Fetch/XHR）、与页面 Hook 的 `postMessage`、菜单消息、`MCP_PROXY_REQUEST` 入口转发 | 调 `mcp-engine.handleMcpProxyRequest`；调 UI 刷新 |
| **`content/ai-analysis.js`** | 调用 AI 分析单条/批量请求 | 经 `background` 转发 LLM；更新 `requestRecords` |
| **`content/mcp-engine.js`** | MCP 工具生成、合并、存储、导入导出、**在页面上下文中 `fetch` 代理调用** | 被 `ui-core` / `mcp-ui` 调用；`core` 调代理 |
| **`content/ui-core.js`** | 主面板、请求列表、筛选、批量操作、详情绑定 | 依赖 `state`、`mcp-engine`、`ai-analysis` |
| **`content/mcp-ui.js`** | MCP 工具列表、日志 Tab、工具栏交互 | 依赖 `mcp-engine`、刷新与 `background` 消息 |
| **`content/isolated.js`** | 启动入口：水合存储、挂拦截、注入样式与悬浮窗/面板 | 聚合初始化流程 |
| **`content/page-hook.js`** | 运行在 **MAIN world**，与页面同源 Hook 协同 | 由 `background` `executeScript` 注入 |
| **`extension/background.js`** | Service Worker：Native Messaging、MCP 调用编排、`AI_CHAT_COMPLETIONS`、存储扫描工具、路径参数拆分 | 连接扩展与 OS 进程、与 content 消息 |
| **`extension/mcp-helper/server.mjs`** | Node：Native Messaging + HTTP MCP + 工具缓存 | 与 `background` 配对的 `com.aireq.mcp_helper` |

**`mcp-server/`**（若投递岗位强调服务端）：可描述为「独立探索的 MCP Server 实现/示例」，使用社区 SDK 与 Web 框架，与扩展内 `mcp-helper` 为**不同部署形态**，便于对比「纯本地 Node」与「扩展内逻辑」边界。

## 5. `mcp-helper` 是什么？能达到什么效果？

### 定义

`mcp-helper` 是一个跑在用户本机上的 **Node 进程**（`extension/mcp-helper/server.mjs`），承担两类角色：

1. **Native Messaging Host**：与 Chrome 扩展 **stdin/stdout** 通信，接收扩展推送的 **工具定义缓存**（`SYNC_TOOLS`），并把 **MCP 客户端发起的工具调用**（`CALL_REQUEST`）转交给扩展执行，最后把 **`CALL_RESULT`** 回传以完成 JSON-RPC 响应。
2. **面向 MCP 客户端的 HTTP 服务**：在本地端口上实现 MCP 所需的 **HTTP/Streamable** 交互（如对 `/mcp` 的识别、JSON-RPC 方法分发），使 **Cursor 等**能把本进程当作 **MCP Server** 连接。

### 达到的效果

- **效果 A**：在 IDE 里能看到与扩展同步的 **tools/list**，像调用本地 API 一样 **tools/call**。  
- **效果 B**：工具真正执行时，可走 **「找同 origin 标签页 → 内容脚本 fetch」**，尽量复用 **用户已登录态**（见下一节鉴权）。  
- **效果 C**：扩展侧可记录 **调用日志**（`MCP_GET_CALL_LOGS`）辅助排障。  
- **可选**：环境变量 **`MCP_AUTH_TOKEN`** 可在 `initialize` 时做简单鉴权，避免本机端口被其它进程误连（实现见 `server.mjs`）。

## 6. 网站接口的鉴权如何解决？是否有跨域情况？

### 6.1 鉴权（本仓库实际策略）

- **样本头复用**：生成工具时，把当时请求上的头信息记入 `_meta`（如 **`rawRequestHeaders` / `sampleRequestHeaders`**）。  
- **执行时**：`background` 组装 `execHeaders` 与 URL、方法，发给内容脚本；`handleMcpProxyRequest` 用 **`fetch(url, { headers })`** 发起调用。  
- **Cookie / Session**：优先路径是 **目标 origin 已打开标签页**，在**该页面的内容脚本**里发 **同源 `fetch`**，浏览器通常按**普通页面请求**规则带上 **Cookie**（实现意图如此；具体以目标站 SameSite、第三方 Cookie 策略为准）。  
- **无匹配标签页时的回退**：`background` 里 **`fallbackFetch`** 在 **Service Worker** 里直接 `fetch`。此时 **不保证**与用户在页面上的登录 Cookie 一致（扩展后台请求与用户会话分离），**容易 401/403**——这是设计上的**显式权衡**，面试应诚实说明。

### 6.2 跨域（CORS）

- **页面内代理模式（`proxyMode: 'tab'`）**：`fetch` 发在**页面所在源**，对「工具元数据里的 `origin` + `pathname`」通常为 **同源**，**不触发浏览器的跨站 CORS 限制**（与正常前端调自家 API 一致）。  
- **后台回退模式（`fallback`）**：扩展 Service Worker 对目标 URL 的 `fetch`，在 MV3 中常享 **`host_permissions`** 带来的能力，**不等同于**页面 CORS，但可能 **无法携带用户 Cookie**，且部分站点仍可能返回被拦截的响应——需结合目标站策略说明。  
- **与 AI 接口的「跨域」**：大模型请求在 **background** 发出，不经过页面 CORS；依赖的是 **API Key** 与厂商网关策略。

### 6.3 面试可补充的「未做但可演进」

- **`chrome.cookies` 显式读取并注入 Cookie**（需权限与隐私评审）。  
- **OAuth 刷新**：短期 token 过期后的刷新流。  
- **仅允许内网模型 / 脱敏后再送 AI**。

## 7. 其它方案能否达到同样效果？请比较。

| 方案 | 能覆盖的能力 | 局限 |
|------|----------------|------|
| **Postman / Insomnia + Collection** | 文档化、分享、部分代码生成 | 难自动绑定「真实页面会话」与最新前端行为；与 MCP/Agent 集成要另做 |
| **HAR 导入** | 离线复现一批请求 | 契约抽象、去重、路径模板、与 IDE 工具生态仍要加工 |
| **Playwright / Puppeteer 脚本** | 端到端自动化、可带浏览器上下文 | 维护成本高，不是「边浏览边沉淀工具」的交互 |
| **Charles / mitmproxy** | 全量抓包、改包 | 偏基础设施；MCP 工具层、AI 语义、扩展内产品化需自建 |
| **后端统一 BFF + 官方 OpenAPI** | 最正规的工程解 | 落地慢；旧系统、第三方页面往往没有 |
| **独立 MCP Server（纯 Node）直接调 API** | 标准、易部署 | 鉴权与 Cookie 往往仍要解决；本方案用**真实标签页**补齐「用户态」 |
| **仅扩展、不启 Native Helper** | 仍可做分析、Mock、生成 JSON | **无法**让外部 MCP 客户端稳定连上（扩展不是对外 HTTP Server） |

**结论话术**：本项目是「**会话真实 + 契约抽取 + MCP 生态**」的折中——用 **Native Messaging + 本地 HTTP** 打通扩展与 IDE，再用 **同源标签页 fetch** 尽量对齐 **用户鉴权**，这是与「纯服务端 MCP」差异化的关键点。

## 附：面试官可能还会问的问题（建议准备）

### A. 安全与合规

- 敏感头、Token 是否会落存储？导出是否脱敏？（代码中有导出时确认删 Authorization/Cookie 等交互）  
- 公司内部使用是否要经过安全评审？如何避免把生产密钥写进 MCP 参数？

### B. Manifest V3 特有问题

- Service Worker **休眠**：长连接与状态恢复策略（Native 端口断开、`PING/PONG` 超时提示等）。  
- **host_permissions**：与最小权限原则的取舍。

### C. 正确性与一致性

- 路径模板、`pathPatternKey`、合并工具时如何做**兼容性校验**以避免误合并？  
- 生成工具**去重**策略：同键跳过 vs 覆盖，产品语义是什么？

### D. 可观测性与测试

- `MCP_TOOL_TEST` 与真实 `tools/call` 路径是否一致？  
- 如何自动化测试：**无头浏览器 + 加载扩展** 成本较高，可坦诚说明仍以手工与日志为主。

### E. 多站点、多 Profile

- 工具按 **`ai_req_mcp_tools_{hostname}`** 分区；跨域多套环境如何管理？是否需要「工作区」概念？

### F. 为什么不让扩展直接开启 HTTP MCP？

- **浏览器安全模型**下扩展并不适合作为任意本机服务端；Chrome 也未提供「扩展监听 TCP」的标准方案给网页访问；故采用 **独立 Node 进程** 承担 HTTP MCP，扩展通过 Native Messaging 做「执行后端」。

### G. AI 分析不准怎么办？

- Prompt 工程、截断导致信息丢失、需人工校对；可强调「分析是辅助，生成工具仍以真实流量为准」。
