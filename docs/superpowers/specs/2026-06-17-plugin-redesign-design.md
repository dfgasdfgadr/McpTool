# AI 请求分析助手插件页面重设计方案

日期：2026-06-17  
阶段：Brainstorming 设计稿  
范围：整个插件浮层，包括请求工作台、MCP 工作台、设置、全局状态反馈

## 1. 背景与目标

当前插件是 Chrome MV3 content script 注入的原生 DOM + CSS 面板。它已经承载请求采集、AI 分析、Mock/改写、MCP 工具生成、工具管理、日志和设置等能力，但界面仍以窄浮层为主，筛选、批量操作、列表、详情和危险操作混在一起，导致高频场景下信息密度高但层级不够清晰。

本次重设计的优先目标是效率，而不是单纯换皮。设计方向已确认采用“暗色专业控制台工作台”：保留悬浮球入口，面板支持默认紧凑浮层和可切换宽屏工作台。视觉上参考高级 DevTools / MCP 控制中心，强调高密度、清晰分区、状态可见和操作路径短。

## 2. Taste-Skill 转译

原 taste-skill 偏向落地页、GSAP、AIDA 和大视觉章节。本项目是浏览器插件工具面板，不适合直接套用落地页结构。因此将其转译为适合插件的原则：

- 宽松但高效的信息架构：避免所有按钮挤在一行，用固定壳层、摘要区、列表区、详情区分层。
- 高级暗色视觉：深色背景、克制的蓝紫强调色、清晰状态色、细边框和微光层级。
- 精密网格：工作区使用稳定双栏或表格布局，避免卡片堆叠造成空洞和视觉跳动。
- 动效克制：使用 CSS transition 完成宽屏切换、详情面板进入、行 hover 和状态变化；不引入 GSAP 等重型依赖。
- 按钮对比明确：主操作、次操作、危险操作和批量操作使用不同颜色和位置规则。

## 3. 整体信息架构

插件仍通过悬浮球打开。打开后的主面板有两种尺寸状态：

- 紧凑浮层：适合快速查看请求、执行单条分析、启动 MCP。
- 宽屏工作台：适合批量分析、筛选、MCP 工具整理、查看 MCP 调用日志、编辑请求调试规则。

统一壳层分三层：

1. 顶部状态栏：产品名、当前站点、请求数、MCP 连接状态、当前模式、宽屏切换、关闭。
2. 左侧模式栏：请求、MCP、设置。紧凑状态可显示短标签，宽屏状态显示完整标签。Mock/调试规则保留在请求详情中，调用日志保留在 MCP 工作台子视图中，不新增独立主页面。
3. 主工作区：每个模式都遵循“上下文摘要 + 主列表/表格 + 详情/操作面板”的模式。

主导航建议从当前顶部三 Tab 调整为左侧模式栏，但主区数量仍保持为请求、MCP、设置三类。顶部保留全局状态而不是承载页面切换，这样宽屏状态下横向空间可留给列表和详情。

## 4. 请求工作台

请求工作台采用“左侧高密度表格 + 右侧详情检查器”的结构。

顶部摘要区展示：

- 当前站点和请求总数。
- 已分析数量、Mock 命中数量、重复请求数量。
- 批量任务进度：分析中、生成 MCP 中、删除重复中。

请求表格字段建议：

- 选择框。
- 方法：GET / POST / PUT / PATCH / DELETE。
- 路径：优先展示 pathname，hover 或详情中展示完整 URL。
- 状态码与耗时。
- AI 状态：未分析、已分析、分析失败。
- Mock 状态：未 Mock、命中规则、仅下一次生效。
- MCP 状态：可生成、已生成、重复跳过。
- 重复标记。

筛选区放在表格上方，分为两行或可折叠：

- 搜索：URL / path / AI 分析内容。
- 状态筛选：Mock、AI、方法、重复请求。
- 分组：平铺、按域名、按 endpoint。

右侧详情检查器固定展示选中请求：

- 请求摘要：方法、路径、状态、耗时、时间。
- AI 分析区：未分析时显示主按钮，已分析时显示结果。
- 调试规则区：修改响应、高级改写、取消规则、刷新生效。
- JSON 区：请求头、请求体、响应头、响应体，使用可折叠代码块。

宽屏状态默认显示详情检查器。紧凑状态下详情可从右侧覆盖进入，避免当前展开行造成列表跳动。

## 5. MCP 工作台

MCP 工作台拆为三个子视图：

- 工具库：管理所有 MCP 工具。
- 调用日志：查看工具调用记录。
- 本地导入导出：读取本地导出的 JSON 工具包。

工具库采用表格 + 右侧详情编辑器：

- 表格列：选择框、工具名、来源站点、方法、路径、风险、启用状态、最近操作。
- 顶部工具条：搜索、站点筛选、启用筛选、风险筛选、分组、导入、导出、合并、删除重复。
- 右侧详情：描述、参数 Schema、风险等级、测试入口、编辑入口。

批量操作只在已选择工具后显示上下文工具条，避免默认状态下按钮过多。危险操作如删除已选、清空替换导入，必须进入统一确认弹窗。

调用日志采用事件表格：

- 时间、工具名、状态、耗时、错误摘要。
- 右侧详情展示参数摘要、代理模式、响应摘要、错误详情。

本地导入导出保持当前能力，但视觉上统一为“目录状态 + 文件表格 + 导入策略按钮”。导入策略需要明确显示：合并跳过、合并覆盖、清空并导入。

## 6. 设置与全局反馈

设置页改为分组卡片：

- AI 配置：API Key、Base URL、模型名称。
- MCP Server 配置：端口、鉴权 Token、启动自动同步。
- 工具生成配置：命名策略、增强 MCP 推断。
- 导入导出配置：MCP 导出目录、本地写入状态。
- 危险区：清除规则、重置配置等不可逆操作。

全局反馈分级：

- Toast：保存成功、跳过重复、单步轻提示。
- 顶部任务条：批量 AI 分析、批量 MCP 生成、导入导出等可持续任务。
- 面板内错误条：MCP 未启动、Native Messaging 失败、AI 配置缺失。
- 确认弹窗：删除、替换、清空、批量危险操作。弹窗必须显示影响范围和数量。

## 7. 状态与数据流

保留现有原生 DOM + CSS 架构，不引入 React、Tailwind 或 GSAP。原因是 content script 需要轻量、隔离、低侵入，并且当前代码已经围绕全局 state 与 DOM 刷新构建。

建议将 UI 状态明确分区：

- `state.ui.layout`：紧凑/宽屏、左侧模式栏折叠状态。
- `state.ui.activeMainMode`：请求、MCP、设置。
- `state.ui.requestTable`：搜索、筛选、分组、选中行、详情面板状态。
- `state.ui.mcpTable`：站点筛选、风险筛选、启用筛选、分组、选中工具、右侧详情。
- `state.ui.settingsDraft`：设置页编辑态，避免输入时直接污染已保存配置。
- `state.ui.taskProgress`：批量分析、生成、导入导出的进度与错误。

请求、MCP 工具、Mock 规则等业务数据仍保留现有结构。重设计不改变底层协议和存储格式，优先降低界面复杂度和刷新副作用。

## 8. 错误处理与危险操作

错误处理按影响范围分级：

- 配置缺失：在相关模块顶部显示说明和配置入口，例如 AI Key 缺失、MCP 未启动。
- 单条失败：在对应行和 toast 中提示，不中断整个列表。
- 批量任务部分失败：顶部任务条显示成功数、失败数，并允许查看失败详情。
- Native Messaging / MCP 连接失败：顶部状态栏显示断开，并在 MCP 工作台显示诊断卡片。

危险操作统一使用自定义确认弹窗，不再依赖浏览器原生 `confirm` 作为长期方案。弹窗内容包含：

- 操作名称。
- 影响对象数量。
- 是否可撤销。
- 推荐备份或导出提示。
- 明确的取消和确认按钮。

## 9. 测试与验收

测试策略以手动回归清单为主，关键纯函数和状态逻辑可补自动化测试。

手动验收路径：

- 请求捕获：页面 fetch / XHR 记录正常出现，搜索、筛选、分组、全选和单选可用。
- 请求详情：AI 分析、Mock 响应、高级改写、取消规则、刷新生效可用。
- MCP 生成：单条、当前视图、已选、全部生成路径可用，重复跳过统计正确。
- MCP 工具库：站点筛选、启用筛选、风险筛选、搜索、合并、删除重复、导入导出可用。
- MCP 调用日志：日志加载、展开详情、错误展示可用。
- 设置：保存配置、读取配置、MCP 导出目录、命名策略、自动同步可用。
- 布局：紧凑浮层和宽屏工作台切换正常，小屏下不出现横向溢出。

建议补测试的逻辑：

- 请求签名与重复判断。
- 请求筛选组合逻辑。
- MCP 工具冲突键判断。
- UI 状态分区的纯数据转换函数。

## 10. 非目标

本次设计不包含：

- 不重写为 React 或其它前端框架。
- 不改变 MCP 协议、Native Messaging、后台消息类型。
- 不修改工具存储格式，除非实现阶段发现必须迁移。
- 不追求落地页式大动效，不引入 GSAP。
- 不做与页面业务无关的品牌官网或宣传页。
