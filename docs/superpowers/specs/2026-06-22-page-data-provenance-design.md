# 页面数据溯源设计

日期：2026-06-22
状态：设计稿，已通过对话确认
范围：页面选中文本数据溯源、候选接口/字段弹窗、字段级 Mock patch
非目标：AI 语义推断、DOM 语义分析、跨环境响应回放、数组批量 patch、自动断言唯一来源

## 1. 背景

当前插件已经具备请求捕获、Flow 录制、请求归属、Mock/响应改写、MCP 工具生成与 Flow Toolset Context。第一版闭环解决了“从真实操作流程沉淀可用 MCP 工具”的问题，但前端调试中仍有一个高频痛点：

```text
页面上看到一个值，但不知道它来自哪个接口字段，也不能快速只改这个字段验证 UI。
```

第二版页面数据溯源聚焦这个问题。用户在页面上选中一个可见文本，插件从当前/最近 Flow 请求和最近请求记录中查找响应 JSON 字段候选，用户确认后生成字段级 Mock patch，只修改命中的 JSON Path。

## 2. 目标

1. 让用户能从页面可见文本快速找到候选接口和 JSON Path。
2. 让候选结果可解释，展示匹配原因和置信度，不宣称唯一正确来源。
3. 让用户确认字段后能一键生成字段级 Mock，只改该字段，不替换整个响应体。
4. 复用现有请求捕获、Flow、Mock rule 与请求详情能力，避免新增独立 Mock 系统。
5. 保持已有整响应 Mock、请求改写、Flow 录制、MCP 工具能力不回退。

## 3. 用户旅程

主路径：

```text
用户选中页面上的一段文本
→ 插件显示小浮层按钮“查来源”
→ 用户点击后打开候选弹窗
→ 插件在当前/最近 Flow 请求与最近请求中搜索响应 JSON 字段
→ 弹窗展示候选接口、JSON Path、当前值、所属 Flow/Step、匹配原因
→ 用户选择一个候选字段
→ 输入新值并选择类型/生效模式
→ 插件生成或更新字段级 Mock patch
→ 用户刷新页面或重放请求，验证 UI 变化
```

辅助入口：

- 候选弹窗提供“跳转请求详情”。
- 未来可在 FLOW/REQ 中展示已确认字段来源记录，但第一版不把它作为主操作路径。

成功标准：

- 用户能选中页面值，快速找到最可能的接口字段。
- 用户能只修改该字段来验证 UI，而不是替换整个响应体。

## 4. 产品范围

第一版包含：

- 页面选中文本后出现“查来源”入口。
- 候选接口/字段弹窗。
- 当前/最近 Flow 优先的候选搜索范围。
- 响应 JSON primitive 字段遍历。
- 精确、包含、规范化文本匹配。
- 候选评分和匹配原因展示。
- 用户确认字段来源记录。
- 字段级 Mock patch 生成与保存。
- patch 未命中、非 JSON、类型解析失败等错误提示。

第一版不包含：

- AI 语义推断。
- DOM 周边语义分析。
- 跨环境响应回放。
- 通配符 JSON Path。
- 数组批量 patch。
- 自动断言唯一来源。

## 5. 数据模型

### 5.1 字段来源存储

新增 Chrome storage key：

```text
ai_req_field_sources_<hostname>
```

字段来源记录：

```js
{
  id: "fieldsrc_xxx",
  hostname: "admin.example.com",
  pageUrl: location.href,
  selectedText: "张三",
  requestId: "req_xxx",
  flowId: "flow_xxx",
  stepId: "step_xxx",
  method: "GET",
  url: "/api/user/detail",
  jsonPath: "$.data.user.name",
  originalValue: "张三",
  lastMockValue: "李四",
  confidence: 0.94,
  matchType: "exact",
  createdAt: 1782090000000,
  updatedAt: 1782090000000
}
```

说明：

- 字段来源记录只保存轻量元信息，不复制完整响应体。
- 匹配仍以 `state.requestRecords` 中的响应体为即时数据源。
- 只有用户确认候选后才保存记录。

### 5.2 Mock rule patch 扩展

复用现有 `mockRules`，在 `response` 下新增 `patches`：

```js
response: {
  bodyEnabled: false,
  body: null,
  patches: [
    {
      id: "patch_xxx",
      enabled: true,
      jsonPath: "$.data.user.name",
      value: "李四",
      valueType: "string",
      sourceId: "fieldsrc_xxx",
      createdAt: 1782090000000
    }
  ]
}
```

规则：

- `response.bodyEnabled === true` 时，保留旧行为，替换整个响应体。
- `response.patches` 存在时，对真实响应体或已改写响应体应用 JSON Path patch。
- patch 只支持单字段路径，例如 `$.data.user.name`、`$.data.list[2].name`。
- 同一路径多个 patch 同时存在时，后创建的 patch 覆盖前一个 patch。
- `once` 继续复用现有规则语义，可用于“仅下一次生效”。

## 6. 匹配逻辑

### 6.1 搜索范围排序

候选请求按优先级排序：

1. 当前正在录制或最近选中的 Flow 请求。
2. 当前页面最近捕获请求。
3. 当前站点其它最近请求。

默认跳过：

- 静态资源。
- 埋点、日志、监控请求。
- AI API 请求。
- 非 JSON 响应。

### 6.2 JSON 遍历

遍历响应 JSON 中的 primitive 值：

- `string`
- `number`
- `boolean`
- `null`

数组项保留完整路径，例如：

```text
$.data.list[2].name
```

### 6.3 匹配类型

匹配类型：

- `exact`：字段值字符串化后等于选中文本。
- `contains`：字段值包含选中文本，或选中文本包含字段值。
- `normalized`：去空白、大小写不敏感、去千分位逗号、轻量数字/日期字符串规范化后匹配。

### 6.4 候选评分

评分因素：

- 精确匹配分最高。
- 当前 Flow 请求加分。
- 当前页面最近请求加分。
- 请求为已验证/核心请求加分。
- 字段路径名包含常见展示字段轻微加分，如 `name`、`title`、`status`、`amount`、`id`。
- 最近请求加分。

返回最多 10 个候选，按分数排序。每个候选展示匹配原因，例如：

```text
精确匹配 · 当前 Flow · 已验证核心请求
```

## 7. UI 设计

### 7.1 页面选中入口

用户选中文本后，插件在选区附近显示小浮层按钮：

```text
查来源
```

约束：

- 未选中文本不展示。
- 选中文本过长时截断用于匹配，并提示“建议选择具体字段值”。
- 插件自身 UI 内的选区不触发入口。

### 7.2 候选弹窗

候选弹窗分三块：

1. 顶部摘要
   - 选中文本。
   - 搜索范围：当前 Flow / 最近请求。
   - 候选数量。
   - 提示：这是候选来源，不保证唯一正确。

2. 候选列表
   - 接口：`GET /api/user/detail`。
   - JSON Path：`$.data.user.name`。
   - 当前值：`张三`。
   - 所属 Flow / Step。
   - 匹配原因。
   - 置信度：高 / 中 / 低。

3. 字段 Mock 编辑区
   - 原值。
   - 新值输入。
   - 类型选择。
   - 生效模式：持久 / 仅下一次。
   - 保存字段 Mock。
   - 跳转请求详情。

## 8. 字段级 Mock 行为

保存流程：

```text
用户确认候选字段
→ 输入新值
→ 解析类型
→ 保存字段来源记录
→ 生成或更新 mockRule.response.patches
→ syncMockRulesToPage()
→ Toast 提示“字段 Mock 已保存，刷新或重放请求验证”
```

类型处理：

- 原值为 `number` / `boolean` 时，默认按原类型解析。
- 用户可切换为 `string`、`number`、`boolean`、`null`、`object`、`array`。
- `object` / `array` 使用 JSON textarea 输入。
- 输入解析失败时不保存，并提示错误。

patch 应用：

- 响应是 JSON 时应用 patch。
- 路径不存在时不阻断请求，记录 warning 并 Toast 提示。
- 响应不是 JSON 时跳过 patch，保留原响应。
- 多 patch 按创建顺序应用。

## 9. 错误处理

- 未选中文本：不展示入口。
- 选中文本过长：截断并提示。
- 没有候选：展示“未找到候选字段”，提供“打开最近请求列表”。
- 多个候选分数接近：不自动选中，要求用户确认。
- 响应体过大：只遍历前 N 个节点或前 N KB，避免页面卡顿。
- 循环/非 JSON 响应：跳过。
- JSON Path patch 未命中：不破坏请求，记录 warning。
- 类型输入错误：不保存 patch。
- 页面刷新后：字段 Mock 规则继续生效，字段来源记录保留。

## 10. 测试与验收

核心验收：

- 选中页面文本后能看到“查来源”入口。
- 点击后能展示候选接口和 JSON Path。
- 精确匹配字段排在前面。
- 候选结果展示匹配原因和置信度。
- 用户确认候选后能生成字段级 Mock patch。
- 刷新或重放请求后，只修改该字段，其它响应字段不变。
- 路径不存在、非 JSON、类型输入错误时有明确提示，不破坏请求。
- 已有整响应 Mock、请求改写、Flow 录制、MCP 工具不回退。

回归测试：

- 请求捕获仍正常。
- Flow 录制和请求归属仍正常。
- 旧的整响应 Mock 仍优先按原规则工作。
- `once` 规则仍能被消费。
- MCP 工具生成和测试不受影响。

## 11. 风险与缓解

### 11.1 候选来源不唯一

风险：页面同一个文本可能出现在多个接口字段中。

缓解：

- 文案始终使用“候选来源”。
- 展示匹配原因与置信度。
- 多个候选接近时不自动选中。

### 11.2 大响应体导致卡顿

风险：遍历大型 JSON 可能阻塞页面。

缓解：

- 限制遍历节点数和文本体积。
- 候选搜索异步分片执行。
- 超限时提示“仅搜索部分响应”。

### 11.3 patch 误伤响应结构

风险：错误 JSON Path 或错误类型可能导致页面异常。

缓解：

- 只支持单字段 path。
- 路径不存在时跳过，不抛出到业务页面。
- 保存前校验类型输入。
- 支持在 Mock 管理中禁用或删除 patch。

### 11.4 与整响应 Mock 冲突

风险：同一接口既有整响应 Mock 又有字段 patch。

缓解：

- 明确优先级：整响应 body 替换先执行，字段 patch 再执行。
- UI 中提示该接口已有整响应 Mock。

后续增强但不进入第一版：

- DOM 周边上下文评分。
- AI 语义字段匹配。
- 字段来源记录在 FLOW/REQ 中聚合展示。

- 页面数据溯源结果沉淀到 MCP Flow Context。
